## Git이란?
파일의 변경 사항을 추적하고 여러 사용자 간에 해당 파일들의 작업들을 조율하기 위한 분산 버전 관리 시스템  

---
## Git을 사용하면 좋은 점  
**👥 1. 파일의 변경 사항을 추적**
`Git` 을 이용하면 한 파일 내에서 일어난 변화들을 전부 `Git` 이라는 시스템을 이용해 기록할 수 있다.

**정확히 언제 파일을 수정했는지**,  **그 때 파일 내에서 어떠한 변화가 있었는지** 기록해주기 때문에 **특정 시점에서 이전 내용을 언제든지 다시 꺼내볼 수 있어요.** 또한 기록은 **로그 형식으로 관리되기 때문에 파일을 여러 개 저장하지 않아도 된다.  

👥 **2.여러 사용자 간에 해당 파일들의 작업들을 조율**
한 파일이 Github(원격)에 올라가 있다고 가정했을 때 여러 명의 사용자가 해당 파일을 각자의 컴퓨터(로컬)에 다운로드한다면 어떨까? **한 파일을 동시에 여러 사용자의 컴퓨터에서 접근할 수 있다.**

Github를 이용해 파일 또는 폴더를 다운받으면 **`해당 파일(또는 폴더)의 기록 또한 사용자의 컴퓨터(로컬)에서 살펴볼 수 있다.`**. **모든 사용자가 완전히 동일한 작업 내용을 공유**할 수 있으니 **원활한 협업이 가능**해진다.

**`여러 사용자 간에 해당 파일들의 작업들을 조율한다는 말`**은 곧 **`여러 사용자가 특정 내용에 대해 접근하고 공동으로 작업할 수 있는 것`**  

**💻 3. 분산 버전 관리 시스템**  
`분산 버전 관리 시스템`이란?

> **Division VCS (분산형 버전 관리 시스템)**
> 
> 
> 중앙에서 관리하고 있던 모든 이력을 가진 저장소 전체를 복사하여 사용자의 컴퓨터로 가져와 사용한다.   
 
Git을 이용하면 **여러 명이 Github(원격)에 있던 동일한 내용을 본인의 컴퓨터로 각각 불러와 개별적으로 작업**할 수 있는데, **파일이 수정되고 기록하는 시점**이 버전이라면, **작업한 내용을 기록해 버전으로서 반영**하는 것은 버전 관리라고 볼 수 있다.

그렇다면 `분산 버전 관리 시스템`은 곧 `여러 명이 한 파일(또는 폴더)에 접근해 개별적으로 작업하고, 작업 결과를 기록으로서 반영하는 시스템`으로 이해해볼 수 있다.

### 🤔 내 파일은 어떻게 관리될까?
> 앞에서 Git은 **`분산 버전 관리 시스템`**이며, 파일의 변경 사항을 추적하고 여러 사용자 간 작업 내용을 조율해준다는 내용을 학습했다.
> 

Git이 파일의 변경 사항을 추적한다는 내용을 말씀드렸는데, 여러 파일들을 한꺼번에 관리해야 한다면 어떨까?

Git은 **한 파일이 아닌 여러 파일도 관리할 수 있도록 저장소(`repository`)라는 개념을 사용**한다. **`작업하는 파일들이 저장되는 폴더` 정도의 개념**으로 이해하시면 쉽다! 

> 파일의 변경 사항들이 추적되고 관리되는 저장소(`repository`)는 보통 `레포`라는 표현을 사용해 부릅니다.
> 
> 
> 각 위치에 있는 저장소는 사용자의 컴퓨터 내에 있는 로컬 저장소(`local repository`), Github에 올라가 있는 원격 저장소(`remote repository`)로 나누어 볼 수 있습니다.
> 
> 앞으로는 편의를 위해 각각을 `로컬 레포`, `원격 레포` 라는 표현을 사용해 설명하도록 할게요!
> 

### 👨‍💻 1. 나의 작업 공간, `Working Directory`

내가 현재 작업하고 있는 공간(폴더)를 워킹 디렉토리(`working directory`)라고 부릅니다.

워킹 디렉토리 내의 파일은 기본적으로 변경 사항이 모두 추적되지만, 그렇지 않은 파일들도 있습니다.

보통 보안 등의 문제로 원격 레포에 올라가지 못하는 토큰, 서버의 BaseURL, 또는 충돌 위험이 있어 사용자 간 공유하지 않는 파일들이 이런 경우에 속합니다. 이 경우 `.gitignore`라는 파일을 생성해 해당 파일들을 모두 넣어주는 방식으로 관리하게 됩니다.

`.gitignore`로 처리된 파일들은 변화가 생겨도 추적되지 않고, 원격 레포에도 올라가지 않습니다. `.gitignore`의 사용법에 대해서는 [🥷 2. `.gitignore` 로 파일 추적되지 않게 관리하기](https://www.notion.so/2-gitignore-1c1b57f4596b81c88d60ed0ecd72278a?pvs=21) 에서 조금 더 자세히 알아보도록 하겠습니다!


### 🔁 2. 내 파일의 변경사항을 반영하는 `Staging Area`

추적되는 파일들(즉, `.gitignore`에 추가되지 않은 파일들)의 상태는 세 개로 나누어 볼 수 있습니다.

> 변경 없음(`unmodified`): **변경되지 않은 파일**을 의미합니다.
> 
> 
> 변경됨(`modified`): 변경된 파일을 의미합니다.
> 
> 스테이지됨(`staged`): 어? 이게 뭐지?
> 

`변경 없음`, `변경됨` 정도는 알겠는데 … `스테이지됨`은 뭘까요?

이 개념을 알기 위해서는 `add`와 `commit`의 개념을 먼저 짚고 넘어가야 합니다.

앞에서 Git은 `버전 관리 시스템`이며, **파일의 변경 사항을 추적한다**고 말씀드렸는데요! 

이 때 파일에 변동이 생기는 모든 시점을 Git에서 기록할 수 있는 건 아닙니다. 사용자가 적절한 시점에서 변동 사항을 기록해주어야 하는데요. 

쉬운 예시를 들어드리자면 게임에서 진행 상황을 저장하는 세이브포인트(또는 체크포인트) 정도로 이해하시면 좋을 것 같습니다.

이렇게 **변경 사항이 생긴 `modified` 상태일 때 체크포인트를 만드는 일**을 `commit`(커밋)이라고 부릅니다. 

근데 만약 여러 파일을 동시에 편집한 상태에서 한 파일만 체크포인트를 만들고 싶다면 어떻게 될까요? 조금 쉽게 예시를 들어보겠습니다.

> 현재 작업 중인 파일 `A`, `B`, `C`가 있습니다.
> 
> 
> 원래는 A 파일만 작업하고 커밋하려고 했는데 … 하다보니 너무 작업이 잘 되는 바람에 폭주기관차처럼 B와 C를 한 50% 정도 더 작업해버렸어요!
> 
> 작업 내용이 너무 많아져서 커밋을 한번 하고 싶은데 B와 C 파일의 작업이 끝나지 않아 오류가 12313개가 발생합니다 🥺
> 
> 작업이 완료된 A 파일만 커밋할 수 있는 방법이 없을까요?
> 

위와 같은 상황처럼 **선택적으로 파일을 커밋하기 위해** `staged` 단계가 존재합니다. 쉽게 말하면 커밋을 위한 준비 단계라고 할 수 있어요! 

`staged`는 `commit`을 위한 준비 단계로, **`modified`된 파일 중 선택적으로 해당 파일을 `stage`해서 해당 파일만 커밋**할 수 있도록 하는 것입니다.

이렇게 **`modified`된 파일을 `stage`하는 것**을 `add`라고 부릅니다. 한꺼번에 모든 파일을 `add`하면 모든 파일이 `staged`가 되고, 선택적으로 `add`하면 선택된 파일만 `staged`가 되겠죠?

이렇게 `staged` 된 상태의 파일들을 `staging area`에 있다고 부릅니다. 

아까 예시를 다시 가져와보면 A 파일은 `staging area`에, B, C 파일은 `working directory`에 존재하고 있을겁니다!


### ☁️ 3. 내 작업이 기록되는 곳, `Local Repository`

그렇다면 이렇게 `staging area`에 올라간 파일들을 기록해야겠죠? 

아까 `staging area`에 올라가 파일이 `staged` 상태가 되었을 때 체크포인트를 만드는 것을 `commit` 이라고 부른다는 내용을 학습했습니다.

이렇게 `commit`을 통해 파일의 변경 사항을 기록하는 곳을 `Local Repository`라고 부릅니다. 

내가 반영하고 싶은 변경 사항을 `add`를 통해 `staged` 상태로 만들고, 이후 `commit`을 통해 최종적으로 `local repository`에 반영하는 것입니다!

이후 로컬 레포의 변경 사항들을 원격 레포에 반영하기 위해서는 `push`라는 명령어를 사용하게 됩니다.
